CygNet Redundancy Editor

The CygNet Redundancy Editor is a configuration tool where you can define the relationships between the servers, hosts, and sites in your redundancy environment. The Redundancy Editor is also where you set up the trigger conditions that will initiate the auto-failover process for each domain in the redundancy environment. Once all redundant relationships are known by the system, you can execute failover between machines, stop and start sites, and allow servers to switch the domains on which they are running, and support high availability across data centers.

The editor can be invoked from the RSM in CygNet Explorer via a content-menu option, Configure RSM Redundancy, in the RSM service pane, or can be added to a CygNet Studio screen.

Changes made to the redundancy definition made in the CygNet Redundancy Editor are audited by the AUD service when the AUDIT_LEVEL_REDUNDANCY keyword is set in the RSM configuration file.

Notes:

The CygNet Redundancy Editor requires the installation of Microsoft .NET Framework, in accordance with the CygNet System Requirements document.

It can also be instantiated within CygNet Studio from any screen as desired.

To Define a Redundancy Definition for your Redundancy Environment

Use the CygNet Redundancy Editor to configure the networks, domains, zones, and auto-failover triggers in your redundancy environment.

  1. Using CygNet Explorer, select the Remote Service Manager (RSM).
  2. Right-click in the white space of the RSM service pane and select Configure RSM Redundancy from the context menu.
  3. The editor opens to the Network page. Configure the following network properties:
  1. Select an existing network or click New network to create a new one.
  2. Configure the Name of the network, whether this network is a Control network in the environment, and whether this network will Replicate from the standby domain.
  3. For networks other than the primary network, specify the Source network from where data is replicated.
  4. Specify a Reporting GNS and configure the GNS event for each GNS type.
  5. Click New network to add another network as required.
  1. Click the Domain tab. Configure the following domain properties:
  1. Select an existing domain or click New domain to create new one.
  2. Configure the name of the Domain, the Network to which it belongs, and its Role (Active, Local Standby, or Data-Center Standby).
  3. Select the Failover trigger that will initiate an automatic failover.

    Note: The triggers must be first created on the Auto-failover page.

  4. Click New domain to add another domain as required.
  1. Click the Zone tab. Configure the following zone properties:
  1. Select an existing zone or click New zone to create new one.
  2. Configure the Name of the zone, the Data center to which it belongs, and the Default domain to which this zone defaults.
  3. Specify all the Redundant RSMs in the domain. This can be done manually or click Auto-detect to retrieve all RSMs in the domain. Delete any RSMs that are not redundant. Click New RSM to add another redundant RSM.
  4. Click New zone to add another zone as required.
  1. Click the Auto-failover tab. Configure the following failover trigger properties:
  1. Select an existing failover trigger or click New trigger to create new one.
  2. Configure the Name of the trigger, and whether the failover is in Test mode.
  3. Specify the Delay (seconds) to wait before the failover initiates.
  4. Specify the interval Period (in seconds) to wait before checking this failover trigger's conditions.
  5. Click New trigger to add another failover trigger as required.
  1. Click Automatic recovery options to mass modify local service recovery options all redundant RSMs.
  1. Click to Enable service recovery.
  2. Click to Include recommended Restart action.
  3. Click to Include recommended Failover action.
  4. Specify a different Minimum runtime.
  5. Click Update to modify the automatic service recovery settings for all selected RSMs.
  1. Click Refresh to update the editor at any time as you work to configure the redundancy definition.
  2. Click Export to export the contents of the Redundancy Editor to an XML file. See To Import or Export a Redundancy Definition below.
  3. Click Import to import the contents of an XML file into the Redundancy Editor.
  4. Click Apply to save the definition and synchronize the redundancy definition across all RSMs in the environment.
  5. Click Close to close the editor.

Note: All RSMs defined in the redundancy definition must be uniquely named.

To Import or Export a Redundancy Definition

You can import the contents of an XML file into the Redundancy Editor or export the contents of the Redundancy Editor to an XML file.

When importing settings all existing settings are cleared out, as the import is not additive. On import, any misconfigurations will be flagged in red, in the same manner as if you entered them via the user interface. Definition validation occurs as normal when trying to save changes made via an import. The XML file can be modified by hand if desired.

  1. Click Export to export the contents of the Redundancy Editor to an XML file.
  2. Specify the file location for the XML file and click Save.
  3. Click Import to import the contents of an XML file into the Redundancy Editor.
  4. Browse to the XML file location and click Open. Any existing settings will be removed.
  5. Check for any misconfigurations and fix as appropriate.
  6. Click Apply to save the relationships. The relationships will be validated as usual.
  7. Click Close to close the editor.
