Converting to CygNet Measurement

Note: The CygNet Gas Measurement Repository (GMR) application is a previously offered CygNet Software product that has been discontinued. Effective with the CygNet v8.5.0 release, GMR applications must be converted to FMS.

Existing Gas Measurement Repository (GMR) systems can be converted to CygNet Measurement (FMS) systems operating in either REPOSITORY mode or FULL mode, as licensed for use and listed in the service configuration file Service Info keywords. Both modes involve a variety of considerations, including configuration of your system and devices, administrative options and templates, data retrieval and editing processes, data import/export requirements, the sending of data to devices or SCADA, and data reporting considerations. See CygNet Measurement Overview for more information about these modes.

This topic describes how to assess your needs before converting to CygNet Measurement, and how to initially configure and set up various system elements to achieve the desired functionality in your new system. Gathering Information

Item Requirements/Values Comments

FMS product
  • Verify your licensed product version, e.g. CygNet Measurement v9.4
  • Verify your licensed product mode, e.g. FULL

Possible CygNet Measurement modes include the following.

  • REPOSITORY
  • FULL

See CygNet Measurement Licensing Requirements for more information.

GMR product
  • Make sure your licensed GMR product version is the same as your licensed FMS product version

If they aren't the same, you may need to complete an upgrade procedure.

See the CygNet Upgrade Procedure for more information.

Database selection

Determine your database needs.

If using the Microsoft SQL Server database, do the following.

  • Verify the name of the Microsoft SQL Server Native Client database provider
  • Verify the name of the external server hosting your FMS database
  • Verify the name of the SQL database created for your FMS Service

If used, the Microsoft SQL Server must meet the requirements listed in the CygNet System Requirements document.

CygNet Measurement can operate utilizing a Microsoft SQL Server database or the FMS internal database. Determine which usage is appropriate for your installation.

See FMS Internal Database for a discussion of the differences between database options.

Workflow
  • Familiarize yourself with general FMS workflow information
  • If you are licensed for a FULL (rather than REPOSITORY) CygNet Measurement system, consider the following desired work flow items
  • Determine what you want your records approval process to be
  • Determine whether multiple levels of approval will be desired for editing data
  • Determine what your requirements will be to close a period
  • Determine how you want edits to be categorized
  • Determine if SCADA users will want to receive finalized measurement data

Although your workflow can be highly customized in CygNet Measurement, some elements are commonly used.

See CygNet Measurement Workflow for more information.

Once work flow determinations are made, you can configure administrative settings to support desired behaviors

Device locations
  • Verify the time zone(s) of your devices
  • Verify the DST status of your devices

Time zone and DST information will be needed by the CygNetTimeZones.xml file.

See Date and Time for more information.

Data sources
  • Identify devices that have FMS polling support
  • Identify devices that currently have only GMR polling support

    • Note: In CygNet Measurement (v8.5.0 and later), devices can have FMS or FMS Legacy (formerly GMR-only) polling support, but not both.

  • Identify device import files or other sources you will be accessing for data

 

Note: Only devices configured with FMS data groups support the "Request New Data" token-based request.

To use FMS polling facilities, existing device template files (DTFs) must be updated to include FMS (or FMS Legacy) data groups.

See Device Template Files and Electronic Flow Measurement Data Items in the Remote Devices section for more information.

Station data aggregation

  • Determine which of your existing devices will operate as station meters, and which will belong to station group(s).

Station data can be aggregated in different ways, which is appropriate to different Node types in FMS:

  • Some stations contain multiple meters that are aggregated to form a single point for data reporting purposes. These stations will be configured as "Station Group Nodes" in FMS.
  • Some stations are single meters that also function as a station for data reporting purposes. These stations will be configured as "Station Meter Nodes" in FMS.

See Creating Gas Meter Nodes and Creating Station Group Nodes for more information.

Production Accounting
  • Determine what export method(s) are desired to supply data to production accounting

Data can be exported from FMS in the following ways:

  • SCADA exports
  • Database exports
  • Normalization View data CSV exports
  • CFX/TFX exports

See Managing CygNet Measurement Data for more information.

Balancing
  • Determine what virtual station groupings will be needed to support balance reporting in FMS

You can set up virtual Station Group Nodes to support desired balance reporting.

See Creating Station Group Nodes for more information.

Auditing
  • Determine if you want to use a separate audit service for storing FMS audit information

Auditing tasks in CygNet Measurement are performed by the FMS service (always) and by the Audit service (optionally), depending on whether you are tracking internal (device) or external (configuration) data.

See Auditing in CygNet Measurement for more information.

Report types
  • Consider what familiar report types you will want to translate into FMS

A variety of standard reports can be produced from FMS.

See Running CygNet Measurement Reports for more information.

SCADA screens
  • Identify any SCADA screens you want to see nested within FMS Explorer

To view SCADA data alongside Measurement data, screens can be accessed via the Vision control.

See Using the Vision Control for more information.

CygNet Studio screens
  • Decide if you want to place CygNet Measurement controls on your CygNet Studio screens

FMS ActiveX controls can be included on CygNet Studio and CygNet Vision screens.

See FMS ActiveX Controls for more information.

Custom Dashboard columns
  • Identify any polling facility UDC values you want to see in the Dashboard control, such as communication statistics

Optional statistics can be configured for display in the Dashboard control.

See Configuring Custom Dashboard Columns for more information.

Replication

  • Determine your replication needs regarding disaster recovery
  • Determine your replication needs regarding business access
  • Determine your redundancy needs regarding failover support

Data synchronization can be set up to replicate several services from one "master" FMS service. Redundancy can be set up for data-center (not local) failover.

See FMS Replication and Data Synchronization for more information.

Configuring Initial Settings

Item Requirements/Values Comments

Installation mode

Verify or configure the licensed mode for your FMS system

  • REPOSITORY mode
  • FULL mode

In the FMS configuration file (CygNet\Services\FMS\fms.cfg) on your host server, check the value of the FMS_MODE service info keyword.

See CygNet Measurement Licensing Requirements for more information.

Devices

Configure devices to return FMS data

  • Install templates
  • Update customized templates

Devices must be configured to return data that can be consumed by CygNet Measurement. This includes EFM (either GMR or FMS) data groups.

See Configuring Remote Devices for CygNet Measurement for more information.

Configure device Nodes

Device-level data is organized via device Nodes.

Device Nodes can be created manually via the Nodes menu or automatically via the Detect New Devices command.

See Creating CygNet Measurement Nodes for more information.

Configure group Nodes and/or virtual Station Group Nodes

Nodes are grouped via group Nodes, and can be aggregated for various business purposes via virtual Station Group Nodes.

Group Nodes can be created manually via the Nodes menu or automatically via the Generate Smart Groups command.

See Creating CygNet Measurement Nodes for more information.

Consider current GMR Detect New Meters arrangements, if any

Note: Detect New Meters criteria do not transfer from GMR to FMS.

Detect New Meters commands do not migrate from GMR to FMS, although previously created meters still exist in the system.

FMS does support a Detect New Devices command. You will be able to configure new Detect New Devices commands and criteria in FMS, to provide functionality similar to that previously used in GMR.

See Running Detect New Device Commands for more information.

Consider current GMR Smart Groups arrangements, if any

Note: Smart Group definitions do not transfer from GMR to FMS.

Smart Group commands do not migrate from GMR to FMS, although previously created groups still exist in the system.

FMS does support a Generate Smart Groups command. You will be able to configure new Generate Smart Group commands and definitions in FMS, to provide functionality similar to that previously used in GMR.

See Running Generate Smart Group Commands for more information.

FMS Toolbox

If you have an existing GMR system, you can use the FMS Toolbox utility to set up a GMR Migration of existing GMR meters and data into FMS.

Note: Alternatively, in CygNet Measurement you can use Detect New Devices to create new FMS device Nodes.

You can migrate existing files from GMR into FMS utilizing the GMR Migration tool from the FMS Toolbox.

See FMS Toolbox, Migrating Files from GMR into FMS, and Detect New Devices for more information.

If you do not have an existing GMR system, but you already have polling facilities set up in the CygNet FAC service, you can use the FMS Toolbox utility to set up an Import into FMS.

Note: Alternatively, in CygNet Measurement you can use Detect New Devices to create new FMS device Nodes.

You can create templates to import .csv files into FMS utilizing the Import tool from the FMS Toolbox.

See FMS Toolbox, Importing Files into FMS, and Detect New Devices for more information.

Commands

Configure Node and System commands as needed

You can set up customized commands in FMS, to be scheduled or run on an ad hoc basis.

This includes the option to set up automatic Node creation commands in FMS. If you have previously used automatic device detection and grouping commands in GMR (Detect New Meters and/or Generate Smart Groups) you can optionally configure new customized FMS Node creation commands in CygNet Measurement (Detect New Devices and/or Generate Smart Groups) to perform similar functions.

See FMS Commands, Detect New Devices, and Generate Smart Groups for more information.

Validation

Configure validation engines as needed

You can set up multiple validation processes in FMS to verify data against a variety of standards. Data failing validation will then generate exceptions, helping you to identify and manage the data anomalies.

See Managing Validation Engines for more information.

Estimation

Configure estimation engines as needed

You can set up multiple estimation routines in FMS to compensate for data that might be missing, late, or of insufficient quality.

See Managing Estimation Engines for more information.

Reports

Configure report templates as needed

  • Ad hoc reports
  • Scheduled reports

A variety of standard reports can be set up to be run on-demand, or on a scheduled basis, in FMS.

See Running CygNet Measurement Reports for more information.

Configuring Replication and Data Synchronization

Item Requirements/Values Comments

Replication

Optionally identify your data replicating source

  • Master service

The Master service (maximum of one) is the designated source for other replicated services, containing complete system data.

See FMS Replication and Data Synchronization for more information.

Optionally identify your replicating disaster recovery service

  • DR (Disaster Recovery) Slave service

The DR Slave (maximum of one) is the replicating disaster recovery service, containing all of the data in the Master service.

See FMS Replication and Data Synchronization for more information.

Optionally identify your replicating business service(s)

  • Business Slave service(s)

Each Business Slave (maximum of three) is a replicating business service, containing a subset of the data in the Master service.

See FMS Replication and Data Synchronization for more information.
Optionally enable data-center redundancy

Redundancy can be set up for data-center (not local) failover.

See FMS Replication and Data Synchronization for more information.
